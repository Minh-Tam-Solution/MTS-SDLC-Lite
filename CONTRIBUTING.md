## ğŸ¯ Ways to Contribute

### 1. Share Success Stories
The most valuable contribution! If MTS SDLC Lite helped you:

1. Create a **Success Story Issue** using the template
2. Share your results: time saved, quality improved, patterns created
3. Get featured in our case study library

**Template**: [Success Story Issue Template](.github/ISSUE_TEMPLATE/success-story.md)

---

### 2. Improve Documentation
Found a typo? Have a clearer way to explain something?

- **Small fixes**: Open a Pull Request directly
- **Large changes**: Open an Issue first to discuss

**Focus areas**:
- Clarifying explanations
- Adding examples
- Fixing typos/grammar
- Improving diagrams/visualizations

---

### 3. Share Patterns & Templates
Created a reusable pattern following the framework?

1. Document it using our format
2. Test it on at least 2 use cases
3. Submit via Pull Request to `03-Templates/`

**Pattern should include**:
- Problem it solves
- When to use it
- Step-by-step guide
- Expected results
- Common pitfalls

---

### 4. Report Issues
Found something that doesn't work or could be better?

1. Check if issue already exists
2. Create a new issue with:
   - Clear title
   - Description of problem
   - Steps to reproduce (if applicable)
   - Your suggestions (optional)

## ğŸ“ Contribution Guidelines

### Code of Conduct
- Be respectful and constructive
- Focus on ideas, not individuals
- Share knowledge generously
- Give credit where due

### Documentation Style
- **Language**: Vietnamese for local audience, English for international (hoáº·c bilingual)
- **Tone**: Practical, action-oriented, founder-friendly
- **Format**: Markdown with clear structure
- **Length**: Concise but complete

### File Naming
- Use descriptive names: `Pattern-Name-Here.md`
- No dates or versions in filenames
- Consistent with existing structure

### Pull Request Process
1. Fork the repository
2. Create a feature branch: `git checkout -b feature/your-feature-name`
3. Make your changes
4. Test thoroughly (if applicable)
5. Commit with clear messages
6. Push to your fork
7. Open a Pull Request with:
   - Clear description of changes
   - Why the change is needed
   - Any relevant issue numbers
